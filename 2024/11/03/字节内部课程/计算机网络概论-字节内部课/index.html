网络结构：网络的网络

1. 区域网络 (Local Area Network, LAN)：

一个简单的LAN可以是一家公司的内部网络，或者是一个大学的校园网络。它通常由一些相互连接的计算机和其他设备组成，如打印机、服务器等。这些设备通常位于同一建筑或同一地区内，使用以太网、Wi-Fi或其他技术进行通信。

例子：一家公司的员工可以使用LAN来共享文件、使用公共打印机，或者从公共服务器获取数据。

2. 城域网络 (Metropolitan Area Network, MAN)：

一个简单的MAN可以是一座城市的公共网络，用于提供公共服务，如公共Wi-Fi、公共安全系统等。它通常涵盖了一个城市或地区的全面，使用各种技术进行通信，如微波、光纤等。

例子：在一座城市，公民可以在公共场合如公园、图书馆等使用公共Wi-Fi。

3. 广域网络 (Wide Area Network, WAN)：

一个简单的WAN可以是一个国家或全球的网络，用于提供各种服务，如Internet、电子邮件、文件传输等。它通常涵盖了全球范围内的多个地区，使用各种技术进行通信，如通信线路、 satellites等。

例子：当你使用Internet时，你的数据实际上是通过一个全球的WAN进行传输的，它将你的数据从你的计算机发送到目标地，无论是在你的国家还是在全球的另一端。

交换

1. 电路交换 (Circuit Switching)：

电路交换是一种通信方式，它在通信之前，先在发送方和接收方之间建立一条物理链路（或“电路”）。这条链路在整个通信过程中都保持活跃，直到通信结束。这种方式类似于打电话：你拨打一个号码，建立一条链路，然后你可以与对方进行通信，直到你挂断电话。

2. 分组交换 (Packet Switching)：

分组交换是一种通信方式，它将数据分成小的部分（或“分组”），然后发送这些分组。每个分组都被发送到目的地，但它们可能会通过不同的路径到达。当所有的分组都到达目的地后，它们会被重新组装成原始的数据。这种方式类似于发送一封分多个部分的邮件：你将一封邮件分成几个部分，每个部分都发送给目的地，然后当所有的部分都到达后，它们会被重新组装成完整的邮件。

3. Web中的网络

HTTP协议

报文可以直接以ASCII形式明文展示，典型的请求-响应模型

  1. 客户端需要等待服务端响应之后才能进行下一步操作，网络利用率不高
  2. 无法在一个链接上进行多路呼叫
  3. 头部信息(Cookie)常常
    会被多次发送

连接模型与队头堵塞问题


Short-lived HTTP1.0 需要多次建立连接

Persistent HTTP1.1 维持一段时间的连接，实现连接重用（持久连接模型）
队头堵塞——当一个请求非常大，第二个很小的请求仍然要等待前一个请求处理完才能进行加入。导致资源浪费

Pipelining 默认情况下，HTTP都是按照顺序进行响应的
允许先发一部分请求过来。虽然看上去很美好，但是很少有浏览器这么干。1.对于队头堵塞一点解决帮助都没有；2.会额外增加一些安全问题

目前方案
同时建立多个连接，将请求通过不同连接进行发送。其可以缓解队头堵塞。
1.HTTP连接并非无条件增长，用户和服务器带宽固定。每个连接分得带宽减少导致单个请求完成时间拉长，所以浏览器会限制连接数（基本上是6个）
2.每次建立连接的成本很高

HTTP3：QUIC

  • Quick UDP Internet Connection
  • 现存网络设备对TCP和UDP支持已经僵化
  • UDP不靠谱但是QUIC靠谱
  • QUIC可以为除HTTP协议以外的应用层协议提供支持

之所以不重新搞一个传输层协议，是因为”惯性积重难返“。所以只好在原有基础上进行更换。

TCP（传输控制协议-Transmission Control Protocol）是一种面向连接的、可靠的、基于字节流的传输层通信协议。它要求设备遵守统一的通信规程，在数据传输前需要建立连接，并在传输过程中采用应答确认机制来确保数据的可靠性和顺序性。类似于打电话，双方在通话前需要先建立连接（拨号），连接建立后才可以开始对话，而且对话的内容是有序的，对方每说完一段话，这边都会有一个回应（应答）。
UDP（用户数据报协议-User Datagram Protocol）是一种无连接的传输层协议，它不需要预先建立连接就可以直接发送数据。UDP尽可能快地传输数据，但是不提供可靠性保证，也不保证数据的顺序。这就像寄明信片，你写好内容后直接扔进邮筒就可以了，不用管对方是否收到，也不用担心明信片的传递顺序。
尽管UDP不如TCP可靠，但它具有实时响应的特点，因此常用于注重速度而可以容忍少量丢包的应用，如视频直播、语音聊天等。
QUIC（快速UDP互联网连接）是Google开发的一种传输层网络协议，位于UDP之上，旨在减少网页加载延迟并提供更快的浏览体验。QUIC改进了包括握手过程在内的多个方面，以加快连接建立的速度，并减少了重传过程中的延时。当某个数据包丢失时，传统的TCP会等待该包的重传，而QUIC则允许其他数据包继续传输，从而提高了效率。目前，QUIC已经发展成为HTTP/3的基础。

CDN：物理极限

CDN-Content Delivery Network（内容分发网络）

相当于每次输入网址，实际上是重定向到了最近的一台厂商所拥有的服务器上。否则每次都要靠多个结点传输数据的话，网络供应商表示”赚麻了“！！！

DNS（Domain Name System）

  • 域名解析一般由网站自己处理
  • 要加速的域名则重定向到 CDN 厂商的域名解析服务处理（未必是地理位置上越近越好）
  • CDN 厂商根据来源确定最近的 CDN 服务器的 IP
  • 用户直接访问最近的 CDN 服务器

保存内容策略(抖音存视频的策略)

拉策略-根据用户访问次数各个节点决定什么内容需要保存

推策略-由总结点决定哪些内容需要保存

例如刚上映的最热门电影，可以先采用推策略使每台CDN服务器都具有一份拷贝。而平时冷门内容，则可以采用拉策略。

小结

  • HTTP 1 2 3 演化历史
  • CDN解决了HTTP协议之外的问题
  • WebSocket从HTTP协议升级而来

4. 网络安全

经过每一台机器，该机器都可以获得该包的内容

网络安全三要素

  • 机密性：攻击者无法获知通信内容
  • 完整性：攻击者对内容进行篡改时能被发现
  • 身份验证：攻击者无法伪装成通信双方的任意一方与另一方通信

1. 机密性

保护秘密信息（加密算法和密钥）

对称加密和非对称加密

  • 对称加密：加密、解密用同样的密钥
  • 非对称加密：加密、解密使用不同的密钥（公钥和私钥），而且公钥加密只能用私钥解密、私钥加密只能用公钥解密。

密码散列函数（哈希函数）

  • 输入：任意长度内容
  • 输出：固定长度内容
  • 重要：两个内容不相同的信息生成相同结果，在计算上是不可能的

如何实现机密性

  • 已知：网络是明文的
  • 如果双方可以通过明文通信商量出秘密信息，那么攻击者也可以
  • 所以想要通过明文通信交换秘密信息，通信双方需要==先有==秘密信息
  • 也就是说，在完全没有前置秘密信息条件下想要在不安全环境下构建机密性是不可能的

2. 完整性和身份验证

二者是相互关联的

  • 蟹老板向银行发起了转账请求
  • 银行需要确认
    • 这个请求真的是蟹老板发起的 
    • 目标账户和转账金额没有被篡改

HTTPS（Hypertext Transfer Protocol Secure）

加密

  • 对称加密：加密和解密都是使用同一个密钥
  • 非对称加密：加密和解密需要使用两个不同的密钥——公钥(public key)和私钥(private key)。可以近似理解为锁和钥匙之间的关系。

小结

  • 网络安全三要素：机密性、完整性和身份验证
  • 在没有提前交换秘密信息的前提下，无法在不安全的信道交换秘密信息
  • PKI保证了普通用户不需要“面对面”和根证书机构交换根证书
  • HTTPS 使用 PKI 完成了除客户端身份验证以外的特性，客户端身份验证靠 HTTP 协议实现
